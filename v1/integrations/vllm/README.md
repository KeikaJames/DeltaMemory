# Mneme × vLLM Integration

This directory contains the **production-grade** integration of
`AttnNativeBank` with [vLLM](https://github.com/vllm-project/vllm).

---

## Architecture

```
vllm.LLM(model=...)
    └─ llm_engine.model_executor.driver_worker.model_runner.model
           └─ (same nn.Module as HF transformers path)
                  │
                  ▼
         AttnNativePatcher  ←── write_fact() / fresh_bank()
                  │
                  ▼
       patched forward() — K/V concat at every attention layer
                  │
                  ▼
         vllm.LLM.generate(prompts, SamplingParams)
```

The key insight: vLLM loads the model weights into a standard `torch.nn.Module`
accessible via the worker's `model_runner`.  By applying `AttnNativePatcher`
to that exact object **before** calling `llm.generate()`, the bank's K/V
injection is live for every token generated by vLLM — no custom vLLM layer
subclassing needed.

The `BankAttachedLLM` wrapper in `bank_attached_llm.py` handles:

1. vLLM `LLM` launch
2. Model unwrap (`_unwrap_vllm_model`)
3. `AttnNativePatcher` + `AttnNativeBank` initialization
4. `write_facts()` / `clear_bank()`
5. Context-managed `generate()` that activates the bank only during the vLLM
   forward pass

## Installation

vLLM is **optional** — `deltamemory` and the rest of the repo import cleanly
without it.  To use this integration on GB10 (spark1):

```bash
# Inside .venv-gb10
pip install vllm
# Verify
python -c "import vllm; print(vllm.__version__)"
```

Tested with: **vLLM ≥ 0.4.x**, torch 2.13, CUDA 13.0, gemma-4-31B-it.

## Quick start

```python
from integrations.vllm import BankAttachedLLM

bllm = BankAttachedLLM(
    model="/home/gabira/Desktop/workspace/models/whitelist/gemma-4-31B-it",
    dtype="bfloat16",
    tensor_parallel_size=1,  # single GB10 GPU
    max_model_len=4096,
    alpha=1.0,
)

# Write facts (same API as HF path)
bllm.write_facts([
    ("city_france", "Paris is the capital of France.", "Paris"),
    ("ceo_apple",   "Tim Cook is the CEO of Apple.",   "Tim Cook"),
])

# Greedy generation with bank active
outputs = bllm.generate(
    ["What is the capital of France?"],
    max_new_tokens=32,
    temperature=0.0,
)
print(outputs[0].outputs[0].text)
# → "Paris."

# Recall@5 probe (top-5 logits at last position)
top5 = bllm.recall_top5("The capital of France is")
print(top5)
# → ['Paris', 'the', 'a', 'Lyon', 'Nice']
```

## What the tests verify

| Test | What it checks |
|---|---|
| `test_model_loads` | BankAttachedLLM unwraps the nn.Module successfully |
| `test_bank_has_facts` | `write_facts()` populates the bank |
| `test_recall_top5` | Target token in top-5 for 3 seeded facts |
| `test_alpha_zero_matches_baseline` | alpha=0 logits match unpatched model (Gate 13A.1) |
| `test_generate_returns_text` | `generate()` produces non-empty text |
| `test_hf_vllm_logit_parity` | HF and vLLM top-1 token agree |

## vLLM version compatibility

| vLLM version | Unwrap path | Status |
|---|---|---|
| ≥ 0.5.x | `llm_engine.model_executor.driver_worker.model_runner.model` | ✅ primary |
| 0.4.x | `llm_engine.driver_worker.model_runner.model` | ✅ fallback |
| Multi-GPU (TP>1) | Ray worker path (see `_UNWRAP_PATHS`) | ⚠️ needs verification |

If your vLLM version uses a different path, add it to `_UNWRAP_PATHS` in
`bank_attached_llm.py`.

## Prior stub (V.1)

The original stub in `examples/vllm_integration/mneme_vllm/logits_processor.py`
implemented a `LogitsProcessor` operating at token-logit level.  That approach
is architecturally sound for logit-space interventions but **cannot implement
AttnNativeBank** because the bank requires access to attention K/V tensors
inside the model forward pass — not just the output logits.  The forward-hook
approach here (model unwrap + `AttnNativePatcher`) is the correct path for
real K/V-bank injection.  The V.1 `LogitsProcessor` stub is retained for
reference and for future logit-space baselines.
