# Preregistration

This page is the binding contract for v3 and v3.1. It was written **before** the held-out test eval was run, in the spirit of Open Science Framework preregistration. Departures from this contract require an amendment, also recorded here.

The full source-of-truth document lives at `docs/preregistration.md` in the repo. This wiki page is a digest plus the running list of amendments.

---

## v3 frozen contract (sealed before Phase G)

**Spec file:** `deltamemory/configs/v3_frozen.yaml`

**SHA-locked artifacts:**

| artifact | sha256 |
|---|---|
| K-projector weights | `be14ae877bc5101a1da5f795d98e0d2ae19bb86156505e46047b53eb162ebd53` |
| `v3_frozen.yaml` | (see commit `430daf8` for the canonical content) |

**Hypotheses:**

- **H1**: v3 recall@1 on the held-out test split > B0 recall@1 (one-sided Wilcoxon, α=0.05).
- **H1b** (decomposition): `v3 − B0 = (v3 − v2) + (v2 − B0)`. We expect both terms to be positive at frozen N.

**Eval split:** `eval/splits/test.jsonl` (N=39 paraphrases, 6 relations, sha-locked at preregistration commit). The split was generated by `scripts/build_lama_loro_splits.py --seed 0` and never re-shuffled.

**Decision rule:** if Wilcoxon p > 0.05, H1 is not rejected; if v3 < B0 with p < 0.05, H1 is **actively rejected** (we report it as a negative result rather than null). No multiple-comparison adjustment for H1 alone, because a single hypothesis is being tested.

**No re-running.** Phase G runs each condition exactly once with seed 0. We do not pick the best of three seeds.

## Result of Phase G (committed)

```
recall@1 (N=39):
  B0 no-memory          = 0.359
  v3 frozen             = 0.278   wilcoxon p = 0.007 (vs B0)
  v3 - v2               = +0.278  (H1b confirmed: projector lifts the bank)
  B1 prompt-insertion   = 0.658
```

**H1 outcome:** rejected. v3 < B0 with p < 0.05 in the wrong direction.

**H1b outcome:** confirmed. The trained K-projector does lift v3 above v2 by +0.278; the issue is that v2 starts at 0.000 (raw bank with no projector is incoherent), so the lift is insufficient to clear B0.

The full report is at `reports/cleanroom/stage14_test_gemma4_e2b/REPORT.md`.

## Methodology amendment (Stage 15)

Because H1 was rejected, we amend the preregistration before running v3.1. The amendments are:

### A1. Structural softmax-dilution fix — `bank_topk`

At N=39, the 39 bank entries each compete with the prompt's ~10 tokens in softmax on equal footing; the softmax distributes mass too thinly. We add a `bank_topk` gate that masks the (N − k) lowest-scoring bank entries with `-inf` before softmax. `bank_topk = 0` (default) is a no-op (bit-equal v3). The gate is shipped already (commit `0036f62`).

### A2. 5× training scale + cross-relation hard negatives

v3 was trained on 520 (write, paraphrase) pairs from 6 relations. v3.1 trains on ≥2,500 pairs from ≥30 relations, with each batch including a fraction of "hard negatives" — fact `f₁`'s K paired with fact `f₂`'s paraphrase from the **same relation** but a **different entity**. This forces the projector to encode entity identity, not just relation surface form.

### A3. Cross-architecture training

v3 was trained on Gemma-4-E2B only. v3.1 trains on a multi-architecture batch: Gemma-4-E2B + Qwen3-4B + (DeepSeek or Llama-3.1-8B). The K-projector is per-layer per-architecture (`k_projector[arch][layer]`); a small router selects the right block at inference based on `patcher.adapter.name`.

### A4. Two-stage val gate

Phase L4 dev sweep selects on `eval/splits/dev.jsonl` (the v3 dev split). After freezing the v3.1 config, we evaluate on a **second held-out split** (`eval/splits/val2.jsonl`, sha-locked at the v3.1 preregistration commit) **before** running test. If val2 recall@1 < B0, we abort the test run and document the outcome — no shopping for a better split.

### A5. Test eval

The Phase G test split (`eval/splits/test.jsonl`) is **frozen for v3 only**. v3.1 evaluates on a fresh test split from the same protocol (`eval/splits/test_v31.jsonl`) so we are not learning anything about test through repeated runs. This is a cost — the v3 test scores cannot be re-compared on a like-for-like basis without a v3-on-test-v31 evaluation, which we will run for fairness.

## What we do NOT amend

- The **red line**: LLM weights remain frozen. No LoRA, no MEMIT-as-our-method.
- The **conservation law**: α=0 / empty bank → bit-equal. Any change breaking this without explicit user opt-in is a bug.
- The **decision rule**: one Wilcoxon, no seed-shopping, single-shot test. We do not retroactively change α=0.05.
- **Open evaluation**: every script + data + report committed to git. No private holdouts.

## Open questions for v3.2

- Bilingual eval (Chinese + English on the same fact set).
- Multi-hop reasoning eval (chain of 2 facts).
- Flash-Attention compatibility (currently we require `attn_implementation="eager"`).
- Bank compression (LoRA-style low-rank K/V to reduce per-fact memory).

These are listed in `docs/wiki/Roadmap-v3.2.md` and are **not** in scope for v3.1.
